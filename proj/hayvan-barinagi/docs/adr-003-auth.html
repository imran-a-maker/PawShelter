Kimlik Doğrulama: JWT + Refresh Token

Stateless API için JWT tabanlı kimlik doğrulama stratejisi

Bağlam (Context)

PawShelter REST API'si stateless bir mimari üzerine kurulmuştur. Kimlik doğrulama için şu gereksinimler belirlenmiştir:

  • Sunucu tarafında oturum durumu (session state) tutulmamalı
  • Rol bazlı yetkilendirme (RBAC): Admin, Kullanıcı rolleri
  • Token süresi dolduğunda kullanıcıyı tekrar giriş yapmaya zorlamadan yenileme
  • Güvenli token saklama ve iletim

Değerlendirilen alternatifler:

  1. JWT + Refresh Token ← Seçilen
  2. Session-based Authentication (Cookie + Server-side session)
  3. OAuth2 (Google, GitHub ile sosyal giriş)
  4. API Key Authentication

Karar (Decision)

JWT (JSON Web Token) access token ve refresh token stratejisi benimsenmiştir. python-jose kütüphanesi ile HS256 algoritması kullanılmaktadır.

Token Akışı (Token Flow)

1
Login: Kullanıcı email + şifre gönderir → API doğrular
2
Token Üretimi: Access Token (30 dk) + Refresh Token (7 gün) döner
3
API İsteği: Her istekte Authorization: Bearer <access_token> header'ı gönderilir
4
Token Yenileme: Access token süresi dolunca refresh token ile yeni access token alınır
5
Logout: Refresh token blacklist'e eklenir (Redis veya DB)

JWT Token Yapısı

Header
{
  "alg": "HS256",
  "typ": "JWT"
}
Payload (Claims)
{
  "sub": "42",
  "email": "user@...",
  "role": "admin",
  "iat": 1710000000,
  "exp": 1710001800
}
Signature
HMACSHA256(
  base64(header) +
  "." +
  base64(payload),
  SECRET_KEY
)

Güvenlik Önlemleri

SECRET_KEY en az 256-bit entropi ile üretilmeli, .env dosyasında saklanmalı
Access token süresi kısa tutulmalı (30 dakika önerilir)
Refresh token HttpOnly cookie olarak saklanmalı (XSS koruması)
Logout'ta refresh token blacklist'e eklenmeli (token revocation)
Tüm API iletişimi HTTPS üzerinden yapılmalı
Şifreler bcrypt (cost factor 12) ile hash'lenmeli

Rol Bazlı Yetkilendirme (RBAC)

Sonuçlar (Consequences)

✅ Avantajlar

  • Stateless: Sunucu tarafında oturum yönetimi gerekmez, yatay ölçekleme kolaylaşır
  • Self-contained: Token içinde kullanıcı bilgileri mevcut, DB sorgusu azalır
  • Standart: RFC 7519 standardı, geniş kütüphane desteği
  • Refresh token ile kullanıcı deneyimi iyileşir

❌ Dezavantajlar

  • Token revocation karmaşık (blacklist gerektirir)
  • Token payload şifrelenmez, hassas veri konulmamalı
  • SECRET_KEY sızarsa tüm tokenlar tehlikeye girer
